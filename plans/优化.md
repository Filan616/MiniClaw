# MiniClaw → OpenClaw-Like 平台化优化计划书

## 0. 目标定位

### 0.1 当前定位

MiniClaw 当前是一个以飞书为入口的个人 AI Agent 执行框架，核心能力包括：

* 飞书 WebSocket 消息入口；
* Gateway 消息路由；
* Agent Loop 工具调用循环；
* DeepSeek/OpenAI/Ollama Provider 抽象；
* ToolRegistry 工具注册；
* PermissionGate 权限网关；
* 路径沙箱与敏感文件保护；
* Shell 黑名单；
* L3 工具审批挂起与恢复；
* SQLite 会话、消息、工具调用、任务、审批等持久化；
* v2 安全增强：事件去重状态机、per-workspace 锁、debug_id 审计、Bypass TTL、ChainDetector、TaskState 上下文保活。

### 0.2 目标定位

本计划的目标不是简单“复刻 OpenClaw”，而是让 MiniClaw 向 OpenClaw 的平台化方向演进：

> 将 MiniClaw 从“飞书单入口个人 Agent Runtime”升级为“多通道、多 Agent、插件化、可审计、可扩展的个人 Agent Gateway”。

最终形态：

* 支持多个 channel：Feishu、CLI、WebChat，后续扩展 Telegram/Slack/Discord；
* 支持多个 agent：每个 agent 独立 workspace、memory、session、tools、skills、sandbox 策略；
* 支持插件化扩展：channel plugin、tool plugin、provider plugin、hook plugin；
* 支持 workspace memory：AGENTS.md、SOUL.md、USER.md、MEMORY.md、daily memory；
* 支持完整 session transcript：append-only JSONL，支持 compaction、pruning、checkpoint；
* 支持 Control UI：查看会话、配置、日志、审批、审计、agent 状态；
* 支持安全审计 CLI：doctor / audit / audit --fix；
* 支持自动化任务：cron、hooks、event trigger；
* 支持更硬的 sandbox backend：host / workspace / docker 三档；
* 保持个人 assistant 信任模型，不宣称支持敌对多租户安全边界。

## 1. 总体架构目标

### 1.1 目标架构图

```text
用户入口
  ├── Feishu Channel
  ├── CLI Channel
  ├── WebChat Channel
  └── Future: Telegram / Slack / Discord
        ↓
Channel Manager
        ↓
Gateway Core
  ├── Routing Engine
  ├── Agent Manager
  ├── Session Manager
  ├── Permission Engine
  ├── Plugin Manager
  ├── Audit Logger
  ├── Config Manager
  └── Control API
        ↓
Agent Runtime
  ├── Agent Loop
  ├── Tool Executor
  ├── ChainDetector
  ├── TaskState
  ├── Memory Manager
  ├── Transcript Manager
  └── Compaction Manager
        ↓
Execution Layer
  ├── Built-in Tools
  ├── Plugin Tools
  ├── MCP Tools
  ├── Shell Runner
  ├── File Tools
  └── Sandbox Backend
        ↓
Storage Layer
  ├── SQLite metadata
  ├── JSONL transcripts
  ├── Workspace files
  ├── Memory markdown files
  └── Audit logs
```

### 1.2 核心设计原则

1. **Gateway 是唯一控制面**

   * 所有 channel、agent、session、审批、审计、配置修改都经过 Gateway。
   * Channel 不直接调用工具。
   * Tool 不直接访问 storage。
   * PermissionGate 只做判断，不做副作用。

2. **Agent 是隔离单元**

   * 每个 agent 有独立 workspace。
   * 每个 agent 有独立 session store。
   * 每个 agent 有独立 memory。
   * 每个 agent 可配置独立 tools、skills、sandbox、model、channel bindings。

3. **Workspace 是记忆和工作目录，不等于硬沙箱**

   * 默认所有相对路径在 workspace 内解析。
   * 绝对路径访问必须经过 sandbox / permission 策略。
   * 真正隔离依赖 sandbox backend，例如 docker。

4. **权限系统分层**

   * PermissionGate：单次工具调用权限。
   * ChainDetector：单 run 内多步攻击链。
   * TaskStateConstraint：任务级软/硬约束。
   * SandboxBackend：运行时隔离。
   * SecurityAudit：事后审计与配置风险扫描。

5. **会话记录 append-only**

   * 原始 transcript 不随便覆盖。
   * compaction 产生新的 summary entry。
   * pruning 只影响模型上下文，不破坏磁盘上的完整记录。

# Phase 0：基线整理与 v2 安全计划落地

## 目标

先把已有 MiniClaw v2 安全计划落地，作为平台化改造的底座。

## 需要完成的能力

### 0.1 事件去重状态机

实现：

* `processed_events` 表；
* `processing / handled / failed` 状态；
* `heartbeat_at`；
* `attempt_count`；
* 后台 heartbeat task；
* 启动时 stale recovery；
* 不在正常消息重试时抢占 processing 事件。

验收标准：

* 服务重启后，重复飞书 event 不会重复执行；
* 长任务运行 10 分钟不会被误判 stale；
* 崩溃后遗留的 processing 事件会在启动时标记 failed；
* failed 事件可重试，并累加 attempt_count。

### 0.2 per-workspace 串行锁

实现：

* `_with_workspace_lock(agent_id, workspace_dir, coro_factory)`；
* `handle_message` 加锁；
* `handle_card_action` 加锁；
* 明确单进程限制；
* 预留未来 DB/file/Redis lock。

验收标准：

* 同 workspace 多消息串行；
* 不同 workspace 消息并发；
* 审批恢复路径不会绕过锁；
* 单元测试覆盖并发写同一文件。

### 0.3 错误三档分级与安全审计

实现：

* `SecurityAuditLogger`；
* `security_audit` 表；
* `Decision.audit_event`；
* debug_id；
* 工具层通过 `ToolContext.audit_logger` 写审计；
* 黑名单、敏感文件、链式攻击进入审计表；
* LLM 只看到模糊错误，不看到匹配规则。

验收标准：

* 黑名单错误不泄露具体 pattern；
* 敏感文件错误不泄露敏感模式；
* debug_id 可在 audit 表中查到完整原因；
* PermissionGate 不直接依赖 storage。

### 0.4 Bypass TTL

实现：

* `/bypass` 默认单次；
* `/bypass next`；
* `/bypass 10m`；
* `/bypass 1h`；
* `/bypass persistent` 二次确认；
* `/bypass confirm`；
* `/safe`；
* `sandbox_mode_single_use`；
* `pending_confirmations`；
* finally 保证单次 bypass 回退。

验收标准：

* 默认 `/bypass` 只影响下一条消息；
* Agent 执行异常时也能自动回退；
* TTL 到期后自动回 safe；
* persistent 需要二次确认。

### 0.5 ChainDetector

实现：

* `permissions/chain_detector.py`；
* `ChainRisk`；
* `evaluate_before_tool`；
* `observe_after_tool`；
* 只在工具成功后记录状态；
* 高危脚本链 deny；
* 普通脚本链 need_approval。

验收标准：

* write_script → chmod → exec 被识别；
* 脚本含 `curl/wget/rm -rf/sudo/.ssh/.env` 时直接 deny；
* 普通测试脚本执行链进入人工审批；
* 失败的 write_file 不会被记录为已写脚本。

### 0.6 TaskState 上下文保活

实现：

* `TaskState`；
* `TaskFact`；
* `FactKind`；
* `task_state` 表；
* `extract_facts_from_messages`；
* compaction summary 落库；
* `get_history()` 手动组装 first_user → summaries → recent；
* `/pin`、`/goal`、`/tasks`、`/compact`。

验收标准：

* 历史压缩后用户约束不丢；
* summary 不排到 recent 后面；
* 多次 summary 超过 3 条会合并；
* recent_errors 从 `agent_runs.messages` 提取；
* TaskState 每轮注入 system prompt。

让 MiniClaw 从“一个 agent_id 的简单路由”升级成真正的多 Agent 管理系统。

```yaml
      - read_file

      - list_directory
      - run_shell
    skills:
      - coding-basic
    memory:
      enabled: true

  list:
    - id: main
      name: Main Assistant
      workspace: ./data/workspaces/main
      channels:
        feishu:
          chat_ids:
            - oc_xxx
      tools:
        allow:
          - read_file
          - write_file
          - list_directory
          - run_shell
      skills:
        allow:
          - coding-basic
          - file-management

    - id: coding
      name: Coding Agent
      workspace: ./data/workspaces/coding
      model: deepseek-coder
      tools:
        allow:
          - read_file
          - write_file
          - run_shell
          - git_status
          - git_diff
          - git_commit
      sandbox: docker
```

## 1.2 新增 AgentManager

新增文件：

```text
mini_claw/agent/manager.py
```

职责：

* 加载 agent 配置；
* 解析 agent defaults；
* 根据 channel/chat_id 路由到 agent；
* 创建 agent workspace；
* 管理 agentDir；

```sql
CREATE TABLE IF NOT EXISTS agents (
    agent_id TEXT PRIMARY KEY,
```

# Phase 3：Plugin 系统

## 目标

让 MiniClaw 从“静态工具注册”升级为“可安装、可启用、可禁用、可审计的插件系统”。

## 3.1 插件类型

支持四类插件：

```text
channel plugin  - 新增消息入口
tool plugin     - 新增工具
provider plugin - 新增模型后端
hook plugin     - 新增生命周期 hook
```

后续再扩展：

```text
skill plugin
memory plugin
sandbox plugin
media plugin
speech plugin
```

## 3.2 插件目录结构

```text
plugins/
  calendar_plugin/
    plugin.yaml
    plugin.py
    README.md

  github_plugin/
    plugin.yaml
    plugin.py
    README.md
```

`plugin.yaml` 示例：

```yaml
id: github
name: GitHub Plugin
version: 0.1.0
type:
  - tool
requires:
  permissions:
    - external_read
    - external_write
entrypoint: plugin.py
tools:
  - github_create_issue
  - github_list_prs
```

## 3.3 PluginManager

新增：

```text
mini_claw/plugins/manager.py
```

职责：

* 扫描 plugins 目录；
* 读取 plugin manifest；
* 校验版本与权限；
* 加载 entrypoint；
* 注册 tools / channels / providers / hooks；
* 支持 enable / disable；
* 写入 plugin registry；
* 提供安全审计信息。

## 3.4 插件 API

```python
class MiniClawPlugin(Protocol):
    id: str
    version: str

    def register_tools(self, registry: ToolRegistry) -> None: ...
    def register_channels(self, manager: ChannelManager) -> None: ...
    def register_providers(self, manager: ProviderManager) -> None: ...
    def register_hooks(self, hooks: HookManager) -> None: ...
```

## 3.5 插件 CLI

```bash
mini-claw plugins list
mini-claw plugins install ./plugins/github_plugin
mini-claw plugins enable github
mini-claw plugins disable github
mini-claw plugins inspect github
mini-claw plugins audit github
```

## 3.6 插件安全策略

第一版不允许自动从网络安装插件，只允许本地目录安装。

安全要求：

* 插件必须声明权限；
* 插件工具默认不自动启用；
* 插件启用需要写入 config；
* 插件加载失败不影响 Gateway 启动；
* 插件不能直接拿 storage，必须通过上下文注入；
* 插件工具必须经过 PermissionGate；
* 插件 manifest 进入 audit。

## 3.7 验收标准

* 可以安装本地插件；
* 插件工具能被 ToolRegistry 发现；
* agent 可以通过 allowlist 控制是否使用插件工具；
* 插件失败不会拖垮 Gateway；
* security audit 能列出插件权限；
* 测试覆盖 plugin scan/load/enable/disable/error isolation。

# Phase 4：Skills 系统

## 目标

引入 OpenClaw 式 skills：让 Agent 不只是有工具，还能有“如何使用工具、如何处理任务”的操作说明。

## 4.1 Skill 定义

Skill 是一个目录：

```text
skills/
  coding-basic/
    skill.yaml
    SKILL.md

  git-workflow/
    skill.yaml
    SKILL.md

  feishu-assistant/
    skill.yaml
    SKILL.md
```

`skill.yaml`：

```yaml
id: git-workflow
name: Git Workflow
description: Help agent inspect diff, create branch, commit safely.
tools:
  recommended:
    - git_status
    - git_diff
    - git_commit
activation:
  keywords:
    - git
    - commit
    - branch
    - pr
```

`SKILL.md`：

```markdown
# Git Workflow Skill

When modifying code:
1. Run git_status first.
2. Read relevant files.
3. Apply minimal diff.
4. Run tests.
5. Show git_diff before commit.
6. Never commit secrets.
```

## 4.2 SkillManager

新增：

```text
mini_claw/skills/manager.py
```

职责：

* 加载全局 skills；
* 加载 per-agent workspace skills；
* 根据 agent allowlist 过滤；
* 根据用户消息和工具上下文激活 skill；
* 把激活的 SKILL.md 注入 system prompt；
* 记录本轮启用的 skills。

## 4.3 Skill 注入策略

注入顺序：

```text
Base system prompt
→ Agent persona
→ Memory summary
→ TaskState
→ Active Skills
→ Tool policy reminder
```

限制：

* 单次最多激活 5 个 skills；
* 总 skill 注入长度默认 8000 字符；
* 超长 skill 需要摘要或拒绝加载；
* group chat 默认不加载 private memory skill。

## 4.4 Skill CLI

```bash
mini-claw skills list
mini-claw skills add ./skills/git-workflow
mini-claw skills enable --agent coding git-workflow
mini-claw skills disable --agent coding git-workflow
mini-claw skills inspect git-workflow
```

## 4.5 验收标准

* agent 可配置 skills allowlist；
* 不同 agent 可启用不同 skills；
* 用户请求 git 相关任务时自动激活 git-workflow；
* skills 不直接赋予工具权限，只提供 prompt 指导；
* 测试覆盖 skill loading、activation、prompt budget、per-agent filtering。

# Phase 6：Session Transcript 与 Compaction 重构

## 目标

把当前 SQLite messages 升级为更完整的 transcript 模型：append-only、可恢复、可分支、可压缩、可审计。

## 6.1 Transcript 存储

每个 agent 的 session 存在：

```text
data/agents/<agent_id>/sessions/
  sessions.json
  <session_id>.jsonl
```

SQLite 仍保留索引：

```sql
CREATE TABLE IF NOT EXISTS session_index (
    session_id TEXT PRIMARY KEY,
    agent_id TEXT,
    chat_id TEXT,
    channel TEXT,
    transcript_path TEXT,
    created_at INTEGER,
    updated_at INTEGER,
    token_estimate INTEGER,
    status TEXT
);
```

## 6.2 Transcript Entry

```json
{
  "id": "entry_001",
  "parentId": null,
  "role": "user",
  "content": "帮我修改 auth.py",
  "createdAt": 1760000000,
  "metadata": {
    "channel": "feishu",
    "agentId": "coding",
    "runId": "run_xxx"
  }
}
```

工具调用：

```json
{
  "id": "entry_002",
  "parentId": "entry_001",
  "role": "assistant",
  "content": "...",
  "tool_calls": [...]
}
```

工具结果：

```json
{
  "id": "entry_003",
  "parentId": "entry_002",
```

## 6.3 TranscriptManager

新增：

```text

```

职责：

* append entry；
* tail read；
* full scan；
* token estimate；
* parent-child consistency；
* tool_call/tool_result pair validation；
* transcript hygiene；
* compaction summary append；

## 6.4 Compaction 改造

保留已有 TaskState，但把 compaction 改成：

```text
Transcript tail read
→ 找到可压缩区间
→ 验证 tool_call/tool_result 成对
→ 提取 facts 到 TaskState
→ 提取 errors
→ 生成 summary entry
→ append summary
→ 更新 session checkpoint metadata
→ 构造 model context 时只取 summary + recent
```

注意：

* 不删除原始 transcript；
* SQLite messages 可逐步降级为索引表；
* compaction 不破坏审计；
* pruning 只影响 prompt，不影响磁盘记录。

## 6.5 验收标准

* transcript 是 append-only；
* 工具调用和工具结果不被拆断；
* compaction 后仍能恢复完整历史；
* get_context 不需要全量读取 transcript；
* 支持 `/session export`；
* 支持 `/session compact`；
* 支持 `/session restore <checkpoint>`。

# Phase 8：Sandbox Backend

## 目标

## 8.1 Sandbox 模式


## 8.2 Sandbox 接口

```python
class SandboxBackend(Protocol):
    name: str

    async def prepare(self, agent, workspace) -> SandboxSession: ...
    async def run_shell(self, command: str, session: SandboxSession) -> ToolResult: ...
    async def read_file(self, path: str, session: SandboxSession) -> ToolResult: ...
    async def write_file(self, path: str, content: str, session: SandboxSession) -> ToolResult: ...
    async def cleanup(self, session: SandboxSession) -> None: ...
```

## 8.3 DockerSandbox

能力：

* 每个 run 创建临时容器；
* workspace mount 到 `/workspace`；
* 可配置只读/读写；
* 网络默认关闭；
* CPU/memory 限制；
* 命令 timeout；
* 非 root 用户；
* 容器结束后清理；
* 支持保留 debug 容器。

## 8.4 配置

```yaml

  default: workspace

  docker:
    enabled: true
    image: python:3.11-slim
    network: none
    memory: 1g
    cpus: 1
    workspace_access: rw
    user: nonroot
```

## 8.5 验收标准

* host/workspace/docker 三种 backend 可切换；

# Phase 15：简历与面试表达定位

## 项目名称

**MiniClaw：面向多通道入口的安全可控个人 Agent Gateway**

## 项目简介

从 0 设计并实现一个参考 OpenClaw 架构思想的个人 AI Agent Gateway，支持飞书/CLI/WebChat 多通道入口、多 Agent 路由、工具调用、权限网关、路径沙箱、事件去重、审批恢复、审计日志、插件化工具扩展、workspace memory、session transcript、上下文压缩与任务约束保活。系统重点解决 LLM 工具调用中的安全性、确定性、可观测性和上下文长期管理问题。

## 技术栈

Python、FastAPI、SQLite、WebSocket、Feishu SDK、OpenAI-compatible API、asyncio、Docker、JSON Schema、Markdown Memory、MCP、pytest。

## 简历要点

* 设计并实现 Gateway 中心编排架构，统一管理 channel、agent、session、tools、permissions、audit 与 workspace。
* 实现 Agent Loop 工具调用闭环，支持多轮 LLM function calling、工具结果回填、重复调用去重、审批挂起与恢复。
* 构建多层安全防御体系，包括路径沙箱、敏感文件检测、shell 黑名单、Bypass TTL、ChainDetector、多级权限审批和 debug_id 审计。
* 实现事件去重状态机与 per-workspace 并发锁，解决消息重试、服务重启和多消息并发写 workspace 导致的重复执行与竞态问题。
* 设计 TaskState + transcript compaction 机制，将长任务中的目标、约束、测试命令、错误信息结构化提升，降低 LLM 长上下文失忆风险。
* 规划并逐步实现 OpenClaw-like 平台能力，包括多 Agent 路由、插件系统、Skills、workspace memory、Control UI、Docker sandbox、security audit CLI 和 MCP 外部工具生态。

# Phase 16：风险控制

## 技术风险

1. **范围过大**

   * 解决：按 Milestone 拆分，先完成安全底座，再做平台化。

2. **插件系统引入供应链风险**

   * 解决：第一版只允许本地插件，不支持远程安装；插件必须声明权限；默认禁用。

3. **Docker sandbox 跨平台复杂**

   * 解决：host/workspace 先保留，docker 作为可选 backend；Windows 走 WSL2。

4. **Transcript 迁移破坏旧数据**

   * 解决：SQLite messages 保留为索引，JSONL transcript 并行写入一段时间，验证稳定后再切换读取路径。

5. **Control UI 暴露风险**

   * 解决：默认 localhost；token 必填；写操作 admin scope；security audit 检查公网暴露。

## 产品风险

1. **功能太像 OpenClaw，缺少自己的重点**

   * 解决：MiniClaw 主打中文/飞书/本地开发场景，强调安全工程和简洁实现。

2. **面试被问“是不是只是抄 OpenClaw”**

   * 回答：不是复刻产品，而是参考 OpenClaw 的 Gateway / multi-agent / plugin / memory / transcript 思路，从 0 设计并实现 Python 版本的个人 Agent Runtime，重点在安全可控工具执行和飞书入口集成。

# 最终结论

本计划将 MiniClaw 的演进分为三层：

1. **安全底座层**

   * 事件去重、并发锁、审计、Bypass TTL、ChainDetector、TaskState。

2. **平台化架构层**

   * 多 Agent、多 Channel、Plugin、Skills、ProviderManager、SandboxBackend。

3. **长期运行层**

   * Memory、Transcript、Compaction、Control UI、Security Audit、Cron、MCP。

完成 Phase 0-2 后，MiniClaw 可以称为“安全可控的多通道个人 Agent Gateway MVP”。

完成 Phase 3-8 后，MiniClaw 可以称为“OpenClaw-like 个人 Agent 平台原型”。

完成 Phase 9-12 后，MiniClaw 才比较接近 OpenClaw 的平台形态，但仍应明确保持个人 assistant 信任模型，不宣称支持敌对多租户隔离。
